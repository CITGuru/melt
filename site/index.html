Cut your Snowflake bill without touching your queries.

Melt is an open-source, self-hosted Rust proxy that sits in front of Snowflake and transparently routes eligible SQL to a cheaper DuckDB-backed lakehouse on your own S3. Per-query. Invisible to the driver. Change one connection string.

The four things Melt does

A drop-in proxy that picks the cheapest correct execution path for every statement.

→ Drop-in proxy

Speaks Snowflake's REST wire protocol exactly. JDBC, Python, dbt, Looker connect unmodified.

⚡ Per-query routing

Every statement gets its own decision: lake, Snowflake passthrough, or hybrid dual execution.

≡ CDC-fed lakehouse

Sync streams Snowflake changes into DuckLake or Iceberg on your own S3. No third party in the path.

◇ Open & self-hosted

Apache-2.0, single binary, runs in a container. No SaaS, no telemetry, no vendor lock-in.

One proxy, three routes

Melt classifies each statement — write detection, Snowflake-feature probe, allowlist and policy gates — and picks one of three execution paths. The decision is invisible to the client.

Run it locally on DuckDB.

Translate the SQL to DuckDB dialect, then execute against the lakehouse copy — Iceberg or DuckLake-backed Parquet on your own S3. Cheap, fast for the small-to-medium analytics that dominate AI-agent and BI traffic.

No warehouse spin-up. No credit burn. No minimum billing window.

When it fires: read-only queries where every referenced table is synced and active, no policy markers apply, and the scan estimate is under [router].lake_max_scan_bytes.

Inspect the decision

$ melt route "SELECT count(*) FROM analytics.events
              WHERE day = current_date"

route:    lake
reason:   eligible read, scan ~12 MB
backend:  ducklake (postgres + s3://melt/lake)
sql:      SELECT count(*) FROM analytics.events
          WHERE day = CURRENT_DATE

Forward verbatim to Snowflake.

Writes, Snowflake-specific functions, policy-protected tables, and anything the lake side can't safely handle goes straight through to your real Snowflake account. Same response shape, same semantics.

Heavy joins and the Snowflake-only long tail stay where they belong.

When it fires: writes (DML / DDL), Snowflake-only features, policy-protected references, oversize scan estimates, and fallbacks from any other classifier failure.

Inspect the decision

$ melt route "MERGE INTO analytics.dim_customer ..."

route:    snowflake
reason:   write detected (MERGE)
upstream: account xy12345.snowflakecomputing.com

Plan-split across both engines.

Some operators run locally on DuckDB, some get pushed to Snowflake; results bridge via Arrow IPC and join in DuckDB. Lifts the all-or-nothing passthrough cliff for queries that touch declared-remote tables, bootstrapping tables, or one oversize table among smaller ones.

Whatever Melt returns equals what Snowflake would have returned — verified by a parity sampler in the background.

When it fires: opt-in per case under [router].hybrid_*. Refuses to federate anything touching policy-protected tables. Every mixed plan surfaces in melt route output and metrics.

Inspect the decision

$ melt route "SELECT u.name, sum(o.amount)
              FROM lake.users u
              JOIN snowflake_only.orders o USING (user_id)
              GROUP BY u.name"

route:    dual
reason:   one declared-remote table (orders)
plan:
  └─ HashAggregate           [duckdb]
     └─ HashJoin             [duckdb]
        ├─ Scan users        [duckdb · lake]
        └─ Arrow IPC bridge  [snowflake → duckdb]
           └─ Scan orders    [snowflake]

Up and running in 60 seconds

No Snowflake credentials required to inspect routing decisions.

# Bring up melt + Postgres (DuckLake catalog) + MinIO (S3)
docker compose up --build

# Inspect routing for any SQL — no infra required
docker compose run --rm melt route "SELECT * FROM analytics.public.events"

# Backend / sync / policy summary
docker compose run --rm melt status

Then point any Snowflake driver at 127.0.0.1:8443:

import snowflake.connector

conn = snowflake.connector.connect(
    user="u", password="p",
    account="xy12345",     # must match [snowflake].account in melt.toml
    host="127.0.0.1",      # ← only thing that changes
    port=8443,
    protocol="http",       # plain HTTP for local dev; TLS in prod
)
conn.cursor().execute("SELECT IFF(1 > 0, 'yes', 'no')").fetchall()
# → [('yes',)]
